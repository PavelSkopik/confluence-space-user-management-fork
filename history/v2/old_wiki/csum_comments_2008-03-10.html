With regards to use with Builder theme, we added extended support for this plugin in Builder 2.0.8. Just add a new link to your menu (eg. Edit > Administration menu) for the user management as follows:

{menuitem}{menulink:customusermanagement}Space user management{menulink}{menuitem}

Gary S. Weaver

We should have a release sometime towards the end of this month (July 2007).

v2.0 of the plugin has been developed by a few of us at Duke University and built upon Rajendra's work.

The current version (pre-v2.0) is currently available at http://svn.atlassian.com/svn/public/contrib/confluence/custom-space-user-management/trunk/. Please note that trunk may be unstable, as there is still some work left to do on it. We will tag it and release it via this page and submit it to the central plugin repository sometime before the end of the month.

The only supported build available for the plugin is Maven 2 can be found in the README in the trunk dir. The Maven 1 build in trunk will be removed in the next few weeks. (Originally was going to keep it but there is no reason to.)

If used with normal "Confluence" user/group management option, the primary outstanding bug in the plugin is one that keeps the page from refreshing after a user is removed from a group. I'm looking into that currently. It has been tested with thousands of users in the Confluence database and uses Atlassian's PagerPaginationSupport and Pager classes to handle a lot of data reasonably well.

The "JIRA" user/group management option has not been tested since v1.1 and has been transitioned for now to a hybrid of using JIRA SOAP service for user/group management with the exception of using a JNDI datasource to read (list) groups from the JIRA database, since that functionality was not available in the SOAP service.

The code needs a lot of cleanup and needs some tests. We also need to remove Duke-specific references and move messages into i18n properties. We also want to change the directory structure to be a "normal" maven 2 layout. We will be updating a lot of it over the next week.

Also note, I would try it in a test environment before putting it into production. The configuration has changed so if you deploy it into an environment where v1.x was deployed, the plugin will not work until you update the configuration (but doing that should be straightforward because it is self-documenting).

Yes, some pretty big changes in v2.0. Basically the changes are:

  • Contains space group creation/deletion (the main reason for the new version, but is optional)
  • Lets you do a search for users by username, fullname, email by starts with, contains, ends with (is optional- if you have a lot of users and don't want to upgrade to new user schema that atlassian support you may find it slow. we wanted to include search by groupname, but would have likely been too slow with a lot of users/groups)
  • UI is almost totally different (bulk import page is almost same though)
  • JIRA functionality uses the JIRA soap service, so it no longer requires the JIRA RPC plugin. However, because of missing functionality around group listing it currently requires the username and password for the JIRA
  • It totally kicks butt
  • It is almost fully i18n'd (except config help comments and config errors) - this wasn't totally altruistic as we need Duke-specific messages and labels, etc.

However, we ran into a few issues that we're still fixing. If we hadn't just gone live with it and had a bunch of clustering issues separate from the plugin, it would have been available by now.

Things left to do before alpha release:

  • Fixing some major bugs that came into full view in a last few days
  • Ren is adding confirm functionality to group removal (has the prototype done using YUI)

Things that we need help from others in testing (mostly because we don't have time at the moment):

  • Testing use of JIRA for user management in plugin
  • Testing LDAP integration

I will do my best to fit in the stuff around making LDAP support authentication, and the info you supplied takes me 95% of the way there (I just don't know how to get info from osuser.xml easily via the API yet, so I may just add that into config for now)

Dan Hardiker

Woah!

Please don't use Base URL unless critically necessary as it results in an absolute URL rather than a relative URL. This can cause all sorts of problems with Confluence sites that can be accessed from multiple entry points – for example every site out there which can run over both HTTPS or HTTP, as well as ones which have multiple subdomains pointed to the same Confluence install.

Instead use the following VTL code in your vm:

<a href="$req.contextPath/admin/plugins/custompermissionsconfig/config.action">$action.getText("error.admin.reconfigure.link.text")</a>

As a general question for the community... would people rather have the Jira SOAP username, password, and endpoint URL set in spaceusergroupmanagementplugin.properties (which you need to put somewhere on classpath on every node running confluence), or would you rather this simply be put into the UI for the configuration?

The only reason we chose to use a properties file, was because we thought there might be a security concern around storing a password in configuration. However, the bad side of using a properties file is that it is a pain to change values and could be a pain to use in a clustered environment since the properties file would have to be the same in all of the nodes (although this is usually a one-time setup unless you have to change the username/password/JIRA SOAP service endpoint URL).

Scott,

The functionality you need is still there. Instead of using regular expressions, you just specify the prefix/suffix (both optional) in the plugin configuration setting: "Group Matching/New Group Name Creation Pattern". The prefix and/or suffix can still contain the dynamic "SPACEKEY". If I understand you correctly, you were just using the csum v1.1 plugin to manage groups with names like "SUFFIX-SPACEKEY-FOO" and "SUFFIX-SPACEKEY-BAR", right? So, what you'd do now is to specify the left textfield of "Group Matching/New Group Name Creation Pattern" to be (without the quotes) "SUFFIX-SPACEKEY" and then leave the suffix field blank. Then it will work exactly as it had before.

The reason we did this was that you can't easily use a regular expression to create new group valid group names that match it. (Regular expressions are quite powerful, so it was unreasonable to try to take a regular expression pattern and a partial name given by the user and then create a valid groupname that not only matched the regular expression, but matched the intent of the regular expression.) When we added the group add functionality, we tried keeping both the regular expression for matching and added the prefix/suffix for group name creation, but that was really confusing to users, because your regular expression and group prefix/suffix had to match (even when we validated this in configuration, it was confusing to users- it just wasn't very intuitive). So we ended up determining that for 99.9% (perhaps 100%) of all cases, people probably didn't use regular expressions in such a way that the prefix/suffix with dynamic SPACEKEY in either/both way of handling group name matching and group name creation wouldn't handle, so we dropped the regex and replaced it with just the group prefix/suffix with dynamic SPACEKEY in either/both.

Hope this makes sense. I was really hoping that this wouldn't affect anyone and that everyone could easily make the migration to the v2.0 way of doing things, which seems like it might work for you per your example.

Thanks!

Gary
