Gary S. Weaver added a comment - 08/Feb/11 10:20 AM

Scott,

The main reason that we haven't added the option you are talking about so far is that mistyping a username is fairly easy to do. The two problems with that are:

1. A username could be given access to a space that shouldn't have it.

2. Deleting users from Confluence is not something to be taken lightly, as it releases all content owned by the user to confluence-users (or at least it did in previous versions of Confluence).

So, basically, it is generally a bad idea to allow space admins to add new users without some validation of the username.

But, I understand why it would be helpful to allow addition of users in some other method other than checking with LDAP to determine if the username is valid.

Is there some mechanism to determine what could be considered a valid username in your setup?

Would it make sense for you to write a class or library that would allow us to interface to it in such a way that users can be validated, and we could integrate that?

The current CSUM configuration area is a mess, to put it lightly. It is just way too much code, and adding any new option takes way longer than it should. If I were you and you are looking for a quick fix, I wouldn't take that on; I'd just hardcode your version of the plugin to ignore LDAP but configure it to use LDAP in the config, or if you have a custom username validation process that others couldn't use, then just write that for yourself. But, we'd much prefer it if you could provide an interface to some username validation source if it would be helpful to others.

The other thing to keep in mind is that the LDAP user creation piece is staring down the barrel of Confluence 3.5 right now. Confluence 3.5 is replacing seraph repos with embedded Crowd, and I don't want to mess with trying to implement the full range of LDAP repo configuration within the current CSUM config even though it is one of our tickets. We use the LDAP validation part of CSUM at Duke, so I feel others' pain in that respect. I've looked at Spring LDAP, and maybe in the end, we make the admin provide a spring config vs. doing it in UI. Don't know yet.

Anyway- let me know,

Gary

Scott Cantor added a comment - 08/Feb/11 10:35 AM

In practice, what you're suggesting is not practical IMHO. In a federated world, you can't know if a username is "valid" unless you're prepared to require people to login first. That's a dealbreaker for most deployments (but obviously if it isn't, the plugin works already for those).

Moreover, even if you do that, you don't fix the possibility of somebody provisioning access for a user with the wrong username. In OSU's case, for example, we could validate (or require prior login) by users and know that smith.100 and smith.101 were both valid. Nothing would prevent a space admin from accidentally adding the wrong smith to a group, particularly if they happen to have similar names (which is certainly possible at a large university).

Ultimately you give space admins power and they are the ones responsible for using it correctly. The alternative is they'll reject SSO and federation, so this is basically an either/or situation.

I take your point about the user creation thing. At this point, it's clear that Atlassian doesn't really grasp the concept of version numbers and API compatibility. It sounds like that's a bigger concern for the SSO plugin, so I'll have to start looking at that.

If I could get the patch accepted if I stubbed out an interface for validating the username, I'd be willing to take a stab at it, but I'm not sure how that would be overridden in practice. If people have to customize the code to support that, we don't really gain anything. Is there a way to inject an implementation of that interface into the plugin now? Obviously you can with Spring, but I don't know how that aligns to the configuration here.

Maybe we could just turn the LDAP option into a class name to enter that supplies the interface?

Gary S. Weaver added a comment - 08/Feb/11 11:07 AM

Scott,

I disagree to the generalization that requiring a valid username is impractical. Where you might not have the ability in Shibboleth to validate a username is a real username, we use LDAP indirectly with Shibboleth and can be fairly sure that a username could potentially be able authenticate via Shibboleth locally and can provide an associated full name and email (which is a point I left out before) so that the space admin can (in-theory) double check that they added the correct user (although that may not happen in larger groups).

I agree to the point that smith.100 and smith.101 would both be valid, but see earlier point about being able to see their full name and email. Also, smuth.100 may not be valid along with a myriad of permutations of smith.100, so there is value in validation, even if it isn't perfect.

I am not so hard on Atlassian. For the most part, they do fairly well, even though I know from version to version that they used to remove/change rather than deprecate. I saw in their official documentation though that their new process (maybe their old one that wasn't always adhered to?) is to not remove support for a deprecated method for at least 2 major revisions. But, that said, I totally understand what you mean. I can tell you for the Confluence Shibboleth Plugin that we'll need to change it to use CrowdService vs. UserAccessor, but I have not gone down the path of Confluence 3.5 yet. It is available, however, and we need to test with it somewhere. I'd do it, but I'd need to coordinate with our Confluence admins and would need the time, which I might not get unless we were upgrading to 3.5, which I don't think we will be soon.

As for the patch, I have no problem just adding the functionality to ignore validation if that is what you want. Unless you wanted to add the option to the config UI (which is nasty to work in- you probably don't want to) then a patch just for that probably wouldn't help much (it is just a matter of ignoring LDAP validation). This is the shortest path I think.

But, if you want to write a class that could validate a username using your system, then if it isn't a totally custom thing to OSU, I'd be happy to just include it in the plugin. I'd say to just use Spring, but other than the @Autowired annotation, we don't have any direct dependency on Spring now, so even though it wouldn't be bad to implement it, we'd need to spend a little time there. Also, I like users configuring in the UI only, and not having to configure via config files (like unfortunately we do now with the Confluence Shibboleth plugin). We could make the CSUM config UI somehow be able to look for classes loaded, but just searching for things in the parent classloaders is not as easy I suspect as it used to be because of Atlassian Plugins and OSGi, but maybe that is just because I need to learn more about it. I'd say that we could add some sort of username validation class/jar upload possibility via the config UI, and serialize and store the class in as a plugin preference, but that is probably too much work also. Just including the class to access your repo in the plugin would be best, if feasible.

Thanks!
Gary
