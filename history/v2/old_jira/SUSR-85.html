Donna,

It appears as though the user tried to specify to use LDAP with atlassian-user.xml, but in the atlassian-user.xml, the whole LDAP section is commented out.

CSUM does support Crowd right now, but what you and the customer were trying to do in the config was incorrect. Basically you configured the plugin to use LDAP, and then didn't give it any LDAP info. What you need to do is to configure the plugin NOT to use LDAP, and it should in theory work correctly.

Gary S. Weaver added a comment - 13/Oct/08 3:38 PM

Donna,

Also, if you could help me come up with some language that makes more sense than what I wrote for the CSUM wiki page, that would really help.

Basically what I was trying to say was:

  • The CSUM plugin doesn't care what you are using for user management, unless you are using Jira for user management.
  • You can use LDAP, Crowd, etc. and it doesn't matter- there is nothing additional to configure and it should in theory just automatically see any users that Confluence lets the plugin see (which should be all of them)
  • IF you are using Jira for user management, you need to configure the plugin to do so. Here's how... blah blah blah
  • IF you are NOT using LDAP for user management (for example, you're using Confluence's built-in atlassian-user or OS User (depending on version of confluence and configuration), but wish to use LDAP for lookup when adding new users to a group (to make sure those users exist and get full name, email info from LDAP), THEN you can order the repositories to put LDAP repository below whatever you want to use (like atlassian-user, osuser, etc. repository) in the list in atlassian-user.xml/osuser.xml and the CSUM plugin will use the LDAP repository defined in the atlassian-user.xml/osuser.xml (even though it isn't a first pick as a repository when authenticating/creating users) for the sole purpose of looking up existance of users and getting users' email and full name from LDAP only when you are using the CSUM plugin to add users to a group that don't already exist or adding users that don't already exist to a group during group creation. The bad side-effect of doing this is that if you list an LDAP repository down as the second repository in atlassian-user.xml/osuser.xml, you've effectively defined another user-repository as a backup repository, which could be a risk, if you don't intend for it to be a secondary (backup) repository for normal user lookup.
  • Do not configure the plugin with LDAP configuration (of atlassian-user or osuser) if you don't have an LDAP repository setup in atlassian-user.xml/osuser.xml, because it won't work.
  • There is no reason you have to use LDAP configuration in order to use CSUM, as long as Confluence already contains all of the users you would want to add to a group.

I'll add this info to the page and as a comment to see if that helps. Hopefully that won't be any more confusing.

Thanks,
Gary

Gary S. Weaver added a comment - 13/Oct/08 4:06 PM

Here was the final wording I used in the wiki:

Here is some information that might help if you're trying to decide how to configure CSUM. Sorry if some of it is a repeat:

  • The CSUM plugin doesn't care what you are using for user management, unless you are using Jira for user management (in which case, you should configure it to use Jira for user management). So feel free to use Crowd, LDAP, etc. The reason for this is that the CSUM plugin uses Confluence's own user API, which basically should (in theory) let the CSUM plugin use whatever Confluence uses for a repository.
  • You don't need to configure the plugin for LDAP unless you want the CSUM plugin to automagically know about users that Confluence doesn't know about when you are trying to add those users to a group (or create a group with those users). If you configure CSUM to use LDAP for this purpose only when it can't find the users via the Confluence API, it will assume that you have an LDAP repository configured (and not commented-out) in atlassian-user.xml/osuser.xml. Note that if you do decide you want to use LDAP as a backup method of adding new users that don't already exist in Confluence when you add them to a group/create a group containing those users then be aware that Confluence itself will try to use the repositories in the order listed in atlassian-user.xml/osuser.xml. So if Confluence already has that LDAP repo listed, even if secondary, then why can't Confluence just see the users in LDAP? It's a good question, and the only answer I can provide at the moment is "don't ask". Maybe someone else can provide a better reason.
  • Do not configure the plugin with LDAP configuration (using atlassian-user/osuser) if you don't have an LDAP repository setup in atlassian-user.xml/osuser.xml, because it won't work.
  • There is no reason you have to use LDAP configuration in order to use CSUM, as long as Confluence already contains all of the users you would want to add to a group.
