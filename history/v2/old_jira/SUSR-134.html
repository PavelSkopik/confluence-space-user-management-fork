Gary S. Weaver added a comment - 09/May/11 4:09 AM

Armin,

Thanks for your feedback, and I'm sorry that this is a problem.

Before I answer your question, could you tell me something about your plugin environment in 3.2? Are you using the Confluence user repository, or are you using LDAP? Are you using LDAP with the plugin as a way to lookup users that the space administrator can add?

Confluence 3.5.x uses embedded Crowd. Although there is a chance I could get this plugin working with the default Confluence user repository in days, since I'm working on another project right now, it would probably be weeks or a few months. The problem is that I ran into issues with operations that CrowdService (the embedded Crowd in the new Confluence) did not do like the old UserAccessor did (some find queries, etc.). This is when I stopped work on it. However, I might be able to use a mix of new and old Confluence API classes to get it to work. I just have not had time yet to implement this, and it is possible that bugs may be found in Confluence that might take a months to fix (or longer).

However, I may not be adding the functionality that allows the plugin to look up users with LDAP prior to their addition, and may replace it with a configuration option that allows the user to attempt to add users that do not exist, even though it would not validate that those users exist. I would have to implement LDAP support in the plugin otherwise, and this would require flexible configuration, and I don't have time to do that, unless I were to easily be able to use Spring LDAP in the plugin. So, this functionality in the new version may either be added in months or possibly never added.

Thanks,
Gary

Gary S. Weaver added a comment - 20/Jun/11 6:30 AM

Appreciate everyone's patience. Spent several hours on this during vacation last week working on it, but there is an issue with it currently that to my knowledge is not being logged properly by the framework being used (xwork), so it will take additional time to figure out. Could be weeks to months before a resolution depending on how much time I can allocate. It almost certainly won't happen this week but others are welcome to take a look at it. Thanks to Rajendra I was able to see some examples I needed to implement a specific usage of crowdService for user addition and then Stefan also suggested using userAccessor.isLicensedToAddMoreUsers() check first, with an implementation posted in https://answers.atlassian.com/questions/746/how-to-create-a-user-with-crowdservice so I'll try to integrate that also.

The two main issues at this point are:

  • (xwork hiding exception issue- so not sure what is wrong): https://answers.atlassian.com/questions/769/how-to-fix-method-dodefault-is-not-defined-in-action-in-confluence-3-5-5
  • Probably need to undo usage of userAccessor in some place that I reimplemented because it checks permissions that space admin does not have. (Probably will need to buy 10 user license so I can get access to source to take a look at implementation in some places.)

Again, anyone who wants to take a look at it is welcome and let me know. I just need to make sure that when I'm working on it that someone else isn't working on it at the same time to ensure we don't step on each other, so just let me know in this ticket that you are working on it, and I'll let you know when I am.

Thanks,
Gary
