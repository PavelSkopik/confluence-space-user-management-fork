Confluence Space User Management Plugin

  • Confluence Space User Management Plugin
  • SUSR-97

When deleteing group that is referenced in another space, the group will not be visible in the 'Manage Users/Groups' tab (sometimes) and the group's view permissions will change (sometimes)

Description

When the space admin tries to delete the group and he receives a message that he does not have deletion privileges (due to references to that group in different spaces), the group will not be deleted (and this is what I expect) and sometimes the space group will not be visible in the 'Manage Users/Groups' tab (not expected) and sometimes the group's view permissions will be changed (not expected).

Use Case:
Space Admin1 creates a group: work-test1 in the Work space. The Work space view permissions are automatically assigned to that new group by the CSUM plug-in.
Space Admin2 creates a reference to the work-test1 group in the 'Friends' space (grants view permissions to the work-test1 group in his space).
And now Space Admin 1 tries to delete the work-test1 group and receives an error message saying that the user has no privileges to delete the group.

When repeating the steps mentioned above, I noticed CSUM plug-in's different behaviors:
1.Group is not deleted, group is visible in the 'Manage Users/Groups' tab, group's permissions did not change
2.Group is not deleted, group is not visible in the 'Manage Users/Groups' tab (even after clicking 'Refresh' button, clearing cache, changing browser), group's permissions did not change
3.Group is not deleted, group is not visible in the 'Manage Users/Groups' tab, group's permissions changed

Gary S. Weaver added a comment - 23/Jun/09 2:46 PM

Karolina,

Sorry to hear that you are having trouble with it.

Here is the deal...

In ConfluenceGroupManagementService.java within our CSUM plugin, we remove all space permissions from a group prior to its deletion as a workaround for an issue with Confluence 2.6.0 ( http://jira.atlassian.com/browse/CONF-9623 ) that was supposedly fixed in Confluence 2.7. If the group deletion fails after that, then we don't attempt to readd permissions.

Since the user does not have permissions to delete the group, the user really should not have rights to remove permissions either, so I think this appears to be a bug with Confluence, so I've logged it as:
http://jira.atlassian.com/browse/CONF-16183

However, I'll add some code to see if I can add permissions back if the delete fails and will provide a patched plugin shortly if that is possible.

Gary
