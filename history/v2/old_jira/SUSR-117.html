Rinku,

Two things for you to check first:

1. After you attempt to delete and you close the browser completely and wait an hour and come back, is the user with the username you deleted still there? I know that at least one large institution, that used OSUser schema sharing with Jira, reported that changes to their repository didn't happen immediately, because of how it was setup. If they waited a little while, the data would reflect the change. They only typically would have to wait seconds to a few minutes at most and click the "refresh" link (not the browser refresh) in CSUM (to reset the cache), and it would display properly. Since CSUM plugin relies on Confluence's user/group management, we have to rely on it to provide the information in a timely fashion, and when it doesn't, there isn't much we can do about it, if that is the case.

2. Can you ensure that user is able to be removed from the repository? For example, perhaps it is a read-only user from an LDAP repository. We may have trouble removing read-only users, in which case we'd need to provide an adequate error message, and we'd need to fix that.

Gary S. Weaver added a comment - 08/Oct/10 6:22 AM

Rinku,

The CSUM plugin needs read/write access via the Confluence API to the repository in order to manage users and/or groups. I can confirm (because we and others use it) that it supports LDAP (at least for now) as a secondary repository used to authenticate that the user is a valid user before adding to a group in the primary repository defined for Confluence (typically a normal Confluence repository, but can be any other than allows read/write via the Confluence API). These repositories up to Confluence 3.5 (not yet out) are defined (in order) in atlassian-user.xml.

If you are not using the normal Confluence repository or another read/write repository (for group member management, in this case), then CSUM is not able to modify the repository, and nothing gets added.

The bug I think that you found is that we must not be checking to see if the thing we are trying to modify/remove (group, user, etc.) is read-only. And perhaps it would be nice if CSUM could manage users in LDAP, but we'd prefer to do that through Confluence/Crowd via the Confluence API, as that ensures we are more compatible with future changes in Confluence. So if the API doesn't support it, we don't.

Hopefully we'll have time to fix this issue soon (check for read-only), however that probably won't help you. If you are unable to use a read/write repository for Confluence, then the CSUM plugin is not going to work for you. I'll add this information to our wiki page also. You might check with Atlassian support and Atlassian documentation about repository configuration as well as your LDAP server documentation/configuration/etc. to see whether there is a way to let Confluence modify LDAP, but I know if it does support that, I'm not aware of it.

Sorry for the trouble,

Gary
