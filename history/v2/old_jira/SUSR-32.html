Description

Trying to manage groups in a Space and receiving this error message:

There are currently no groups with rights to view space 'AeroVPD' and matching the group name pattern 'Wiki-AeroVPD-.*' defined for this space in the plugin configuration.

Please try adding some groups.

However, groups do exist, as seen in this snippet from the Administer Groups option in the main console:

Wiki-AeroVPD-EPIC_Transformation Remove
Wiki-AeroVPD-HTLP Remove
Wiki-AeroVPD-Leadership Remove
Wiki-AeroVPD-Software Development Paradigm Remove
Wiki-AeroVPD-Tempe PCPL Remove
Wiki-AeroVPD-Tempe PCPL Chiefs Remove
Wiki-AeroVPD-Users Remove

Here's our configuration:

Current Settings for Custom Space User Management Plugin:
User Manager Location for Wiki: CONFLUENCE
LDAP Authentication Used?: YES
Provider Type: ATLASSIAN-USER
LDAP User ID Attribute: uid
LDAP Email Attribute: mail
LDAP First Name Attribute: hwnickname
LDAP Last Name Attribute: sn
LDAP Provider Class:
JIRA SOAP Service Endpoint URL: -
JIRA SOAP Service Username: -
JIRA SOAP Service Password: -
Usergroups Matching Pattern: Wiki-SPACEKEY-.*
Group Actions Permitted: NO
New Group Name Creation Pattern: SPACEKEY- (group identifier)
Max Number of UserIds to Act On at One Time: 20
Max Number of GroupIds to Act On at One Time: 1
User Search Enabled: NO
personal.space.allowed: NO

Because of this I can't fully test the LDAP piece as I'm never given an opportunity to change a group!

Oh, and I'm fully prepared for a "dude, this is user error, you forgot to do xyz" type of response, as when in doubt on new options I just said no.

Gary S. Weaver added a comment - 16/Aug/07 1:56 PM

Hey, need to clarify this as it certainly isn't clear in configuration...

The Group Name Creation Pattern does NOT have to be lowercase. Only the Usergroups Matching Pattern, and of that, the SPACEKEY part is not included.

This should be perfectly valid:

Usergroups Matching Pattern: wiki-SPACEKEY-.*
New Group Name Creation Pattern: Wiki-SPACEKEY-

The reason I wrote in the config that Usergroups Matching Pattern must be in lowercase is that it is a regular expression that will be matched against the lowercase version of the groupname. Since I didn't have an easy way of converting a regular expression to support this, I had no option if we were going to keep allowing regular expressions but to just try to tell the user this in the configuration.

I think this is a good reason to NOT support regular expressions personally. I think what we should do is to only allow the user to specify prefix and/or suffix, and that in itself will be the matching pattern regardless of whether you are supporting group creation or not.

I'll add this as a comment into the documentation also... does this answer your questions?
