Gary S. Weaver added a comment - 15/May/08 7:59 AM - edited

Aditee,

Raju talked to Atlassian and determined that http://studio.plugins.atlassian.com/browse/SUSR-74 was a result of an Atlassian bug in the User (and Groups) API in Confluence 2.7.3.

See David Pickering's comment in that ticket:

David Pickering - 13/May/08 07:30 PM
So this was 99% verified to work at Cisco for Confluence, we are still doing testing but everything so far is good.

Between the rebuilt JAR, and applying the patch Rajendra pointed us to http://jira.atlassian.com/browse/CONF-10645 , this works for Conf 2.7.3.

Thank you very much.

Dave

The other option that Rajendra (Raju) said worked for him for that ticket is to upgrade to Confluence 2.8.0.

As for the specific subject of the current ticket though, the way to get the most current data (non-cached) from Confluence is to click on the refresh link (not in the browser, but in the CSUM plugin UI). For performance reasons, the plugin caches user and group data. If we removed that caching, performance would likely significantly decrease and it would complicate the paging process.

Except for the fact that the CSUM plugin caches users and groups (and you can reset that cache by clicking on the refresh link within the plugin UI - that cache is user specific, so closing and reopening the browser also may work, the CSUM plugin uses Atlassian's User (Java) API and Atlassian's Jira SOAP API for all user and group actions, so any issue with groups being added to Jira not getting reflected in Confluence (outside of the CSUM plugin caching I described) is something that you would need to test out in the administrative area of Confluence, and if you could see it was a problem there also, you'd want to add a support issue for Atlassian.

Hope this helps!
Gary

Gary S. Weaver added a comment - 16/May/08 10:20 PM

Aditee,

The CSUM plugin uses the confluence API to get users and groups so it should not be any different than Confluence's administration page with the exception of its caching, but if you either click the refresh link in the CSUM tab, and at worst, close the browser or do whatever you need to do to kill the session (because the cache is only session cache for that particular user).

Unfortunately, I am not available right now for any additional support of the plugin beyond that information (because I'm dedicated to another project, and setting up Confluence 2.7.3 with the patches mentioned in the other ticket or preferably 2.8.0 and testing CSUM in those would take time that I don't have at the moment, although there is a good chance that the problem that you are having is something that Raju noticed before also which is that it takes some time for Confluence to reflect that Jira added a user/group and you must use the refresh link (not the browser refresh) in the CSUM tab to refresh its session cache, or kill the browser (completely) so that it will call the Confluence API again to get the updated groups/users from Confluence.

I would not focus on the plugin so much and I would focus on the Confluence User API. There is a page in the Atlassian wiki that tells you how to turn on debug logging for the Confluence User API. If you can prove that the API is not returning the groups/users that you added even after clicking the refresh link (not the browser refresh) in the CSUM tab, and doing that several times over the course of an hour or so after adding them (just in case there is a delay that Raju had told me about earlier in the Cisco environment) then you should take that concern to Atlassian support.

The CSUM plugin is basically only a faceplate on top of the Confluence API. There is not a whole lot there except UI. All of the paging, querying, etc. is all done in Confluence, not the CSUM plugin. So, the place to focus is on the Confluence User API and Confluence.

Please let me know if I can answer any other questions.

Gary
