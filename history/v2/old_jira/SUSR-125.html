Confluence Space User Management Plugin

  • Confluence Space User Management Plugin
  • SUSR-125

Ability to export groups & group permissions along with space export

Description

It would be helpful (though I am not sure technologically feasible) to be able to export the groups and group permissions for a given space so that space migration of individual spaces between Confluence instances could be possible. Currently, when a space is exported (xml format) none of that spaces groups or group permissions are exported along with it. Is there any simple way to do this currently? Not having this ability makes it difficult to bring up a new instance of Confluence and move individual spaces given that none of the heavily relied upon groups and group permissions within those spaces are able to be migrated.

Gary S. Weaver added a comment - 25/Jan/11 11:32 AM - edited

Billy,

I'd like to leave this as a requested feature, but unfortunately, I don't think this would be a feature for this plugin.

The reason is that this plugin's job in life is to manage a set of groups and users that have access to the space. (Not all groups that have access, but just the ones matching the spacekey name "pattern" defined in the CSUM plugin configuration.)

However, you could easily write a plugin or script to do what you are saying.

Basically, what this plugin does is to create regular confluence groups that have the spacekey in part of the name and gives those groups access to the space. Everything is done using the Confluence API, so there is nothing stopping anyone from writing something to manage those groups.

Am assuming you'd do this:

1. Write a plugin/script that would use Confluence API to get a list of the groups in Confluence that match the spacekey "pattern" specified in the CSUM plugin. Perhaps instead of using the CSUM's config, you require the admin to just copy and paste that "pattern" into whatever interface that you devise.

2. It also should list memberships for each group.

3. It should create some sort of export file with this information. I'd say version the file, but so far, Confluence hasn't really made much of a change in the basic info for groups and users. But maybe version it, just in case.

4. Allow it to read this export file on another server with the same admin plugin, if is same version of file (same version of Confluence really not important for the most part).

5. Attempt to create groups for each of these groups, replacing the spacekey of the old group with the spacekey of the new one.

6. This is the hard part. You need to add membership from the old groups to the new one, but on the new instance, are you sure that the same username = same user? Hope so, otherwise you are giving rights to data to the wrong user. What if the user doesn't exist? Should it be created with the same username? Are we ok not creating the email or fullname or should that be part of the export? Should we set the password to null and hope for the best that the new server is using an alternative authentication mechanism, because you can't get password from the user object via the Confluence API on the original server?

The other things to think about are:

1. Confluence 3.5/Embedded Crowd

In Confluence 3.5, Confluence will be using embedded Crowd. This could mean that groups and users will be shared across various instances in future versions of Confluence, so this may become a non-issue in future versions. So any work put towards the goal in this ticket may not be used for very long.

2. If you are worried about copying groups from one instance to another, you might be getting big enough to have other problems that should be addressed to make things more manageable.

There are large institutions using CSUM, and there is nothing wrong with that. But, if you get to the point of having groups that you need to delegate creation of, and then want to copy those groups from one instance to another, odds are that your organization is big enough to benefit from a shared system that manages groups. That way, those groups can be used for other things as well. That isn't to say that there isn't benefit from having the ability to create groups specifically for Confluence content. But, it is more often the case that a group that needs access to one thing may need access to something else, so why not centralize it? However, assuming that you start using Crowd, and Crowd can integrate with any other group/user management solution you have, then integration of CSUM with crowd in Confluence 3.5 (once we do that) may allow you to have your space managers manage groups not only across Confluence instances, but across other things as well. The naming convention of those groups might be a little screwy though, because they would contain the spacekey. Instead of using the spacekey naming convention, maybe CSUM should start requiring admins to define the groups that a space can manage, etc. That way, the same group could be used across spaces. But, then why not give them access to Crowd or whatever else you are using to manage groups?

(I'm resolving this ticket, because this work should be done outside of the project, but feel free to continue to comment.)

Gary

Gary S. Weaver added a comment - 25/Jan/11 11:46 AM - edited

Ok, given more thought, I'm reopening the issue.

Since the space admin has rights to create groups via this plugin, I could see where you'd want to import a list of groups, transform the names, and then attempt to create those groups.

However, as stated in last comment:

  • I think that the complexity involved is high enough where it wouldn't be implemented easily.
  • Giving the wrong username access to the space if the same username is not the same person in the new space would be bad, and I'm not sure if you could force the admin to confirm each username because in some environments there are thousands of users in a group.
  • One alternative to doing the work defined in the ticket might be to just use Crowd, as it would be available in 3.5. This might solve the issue perhaps of sharing the same group to the same space across instances. While it doesn't solve the issue of sharing the same group among two different spaces in the same instance (or two different spacekeys), that would be a separate issue and would require a lot more rework (as well as being better done elsewhere).

Basically, I'm not sure this ticket will have much traction.

We'll see, though.

I believe that it checks for users in the repositories in the order listed in atlassian-user.xml, or at least that is what I found out from Atlassian in the past, but that was some time ago.

Note that if you are using LDAP for additional groups, then the space admin could currently go into the permissions area to assign space permissions to that LDAP group (without even using the plugin). This might suffice for some groups and may be better than managing groups basically in two places using CSUM (by the space admin) and LDAP (possibly by an LDAP admin).

Perhaps the Manage Users/Groups tab (the CSUM plugin) should show the read-only groups as well that have access to the space, but would not allow you to manage those groups. Then, when you export for import into another space, it would include the permissions to those groups also that the space admin is not allowed to manage. Then when you reimported into the other confluence instance, if that group wasn't there, it was just list it as a warning after the import (for example, "Couldn't assign permission to some_other_department because that group was not found, and you don't have rights to create or administer that group.").

Let me know if that sounds right.

Gary S. Weaver added a comment - 26/Jan/11 8:53 AM - edited

Billy,

Yes, and to be clear, the LDAP was really just a separate topic. Basically I meant that if you do use LDAP, Crowd, Active Directory, etc. etc. for group management, then managing what is essentially the same group (same group membership) in two places and calling them two different names (one in LDAP and one in Confluence via CSUM) is not efficient, unless it is just to weighty of a process for users to manage groups that way (because of tech limitations, politics...).

The exporting thing was about making sure that "other" groups (with permissions to the space that don't match the spacekey pattern) have those permissions propagated to the new server during import, if possible. We'd not been showing these "other" groups in CSUM, and I think that causes confusion. Possibly we should at least list those groups on the main CSUM page somewhere, even though there is a link already to the security page that shows them. That way, if we export those permissions, it wouldn't be a surprise. Exporting/importing these "other" groups' permissions is more than I really feel responsibility for, since those permissions are managed by Confluence and not by this plugin. But, somebody would need to do it. Really, all of this should be part of Confluence's space export, IMO, but that would mean having them integrate CSUM into Confluence, and I think, instead, Atlassian might rather have Crowd be the place that handles delegated group management. However, CSUM does something that Crowd probably never will, which is to skip the step of delegating group admin rights to the space admins, and instead just give that responsibility as part of the responsibility of space administration. But CSUM can also bring the pain and inefficiency of inevitable duplication of groups in organizations where there is a separate layer of group management.

Gary
