Gary S. Weaver added a comment - 26/Sep/07 8:42 AM

Andy,

I'll do that, and let you know, but I have a couple of concerns around that (in a few of my last jobs as a java developer I wrote and helped maintain heavily (at times TOO heavily) multithreaded web services and applications):

1) Best practice is for java libraries to support multithreaded access, so this will need to be fixed in the library as well I'd think.

2) It is best not to synchronize a whole method but only the part of an operation that needs to be synchronized, which would have to occur in the ldaputil. This means that "synchronized" would need to be pulled out of csum plugin when ldaputil is fixed.

3) It is best not to have to synchronize at all if possible, and if this is happening because of a shared static/singleton parsing instance that is being managed by ldaputil, then it would be better to much better to create a new instance of it for each parsing than it would be to lock threads.

4) If this is an issue with digester or atlassian, we should take it up with them.

I'd be more than happy to assist with this issue if you'd like (I'd promise to stay hands off all the code except that which was absolutely necessary to change).

TIA and take me with a grain of salt as always,

Gary

Gary S. Weaver added a comment - 26/Sep/07 9:18 AM

Fix applied to synchronize in csum plugin:

public class LDAPHelper
{
...
public static LDAPUser getLDAPUser(CustomPermissionConfigurable config, String userid) throws ParserConfigurationException,
LDAPException
{
Properties builtConfig=buildConfig(config);
LDAPUser u = null;
// this lock is on a static object (this class, not the instance) so that it is JVM-wide
// error this is intended to fix: "SAX problem: FWK005 parse may not be called while parsing."
// error case: http://studio.plugins.atlassian.com/browse/SUSR-58
// TODO: ldaputil or the libraries it uses need to be fixed to avoid locking like this
synchronized (LDAPHelper.class) { LDAPLookupUtil lookup = new LDAPLookupUtil(builtConfig); u = lookup.getUserDetails(userid); }

return u;
}
...

Committed in revision 10548

will close ticket, but would suggest requesting a Jira project for the ldaputil plugin by emailing developer-relations@atlassian.com and asking for it, and then adding this as a ticket to fix for next revision of ldaputil, and then going back into csum and undoing the synchronized block after is fixed.

Way to reproduce bug in older csum version and see it fixed in trunk version is by enabling LDAP user creation in the plugin and then adding a bunch of users to groups in each window.
