Description

Hi,

Confluence 3.5 is due to ship with Embedded Crowd, which will improve LDAP connectivity with Confluence. It will change how user management is handled, and since your plugin works mainly with the current user management system, we've done some testing of it against the current state of Confluence/Embedded Crowd. It seems to be mainly fine, but there are some minor issues that have been found.

We've reported our testing for you. If you don't have access to it, please give us your login name for confluence.atlassian.com and we will grant you permissions. Also, if you would like a copy of the current state of Confluence with Embedded Crowd to develop against, please let us know and we'll arrange that for you.

Thanks,
Anna
Confluence Developer, Atlassian

Gary S. Weaver added a comment - 06/Oct/10 7:09 AM

Anna,

Hope you don't mind that I'm copying and pasting that page into the ticket so I can respond and other team members can see it:

  • Seems to (mainly) work fine

Would you mind creating bugs in CSUM plugin Jira for things that don't work that aren't documented as CSUM bugs already, and could you comment on and watch an existing bugs that you are speaking of? I'd like to make sure everything is documented that needs to be done. Thanks!

  • Functionality tested (both for LDAP and non-LDAP directories):
    o Via the plugin's space admin screen:
    + Creating groups
    + Adding members to groups
    + Deleting members from groups
    + Searching for users
    o Via the Confluence admin page:
    + Editing space/global permissions for groups created through CSUM
  • Needs Fixing: There's an option in the plugin configuration screen that allows you to define some LDAP settings. It relies on atlassian-user.xml existing, so it will be broken in 3.5+. It supposedly allows for you to add users to groups who are defined in your (Confluence-connected) LDAP directory but don't have a Confluence account. I'm still struggling to find an example of where you would have an LDAP directory hooked up that doesn't add all its users to Confluence though. Apparently the plugin writers are at a bit of a loss too. From their documentation:

'You don't need to configure the plugin for LDAP unless you want the CSUM plugin to automagically know about users that Confluence doesn't know about when you are trying to add those users to a group (or create a group with those users). If you configure CSUM to use LDAP for this purpose only when it can't find the users via the Confluence API, it will assume that you have an LDAP repository configured (and not commented-out) in atlassian-user.xml/osuser.xml. Note that if you do decide you want to use LDAP as a backup method of adding new users that don't already exist in Confluence when you add them to a group/create a group containing those users then be aware that Confluence itself will try to use the repositories in the order listed in atlassian-user.xml/osuser.xml. So if Confluence already has that LDAP repo listed, even if secondary, then why can't Confluence just see the users in LDAP? It's a good question, and the only answer I can provide at the moment is "don't ask". Maybe someone else can provide a better reason.'

Whatever it did, it sounds like embedded crowd will do it anyway. The functionality and option on the config screen can probably be removed all together.

Anna,

What it did was to allow you to have a secondary LDAP repository defined in atlassian-user.xml that was queryable via Andy Brook's ldap-util library provided in the Atlassian community plugins area to validate that a username was a valid LDAP entry prior to addition to the primary repo defined in atlassian-user.xml. There are a number of companies/organizations using this functionality in their wiki, so if it needs to be replaced, we would welcome a suggested method to allow similar functionality in Confluence 3.5, or we'll just have to stop supporting Confluence pre-3.5 versions in later versions of the CSUM plugin. We've done everything we could with time allotted to the project to ensure that the plugin continues to work with the various versions of Confluence.

  • Minor problem: Currently, if you have your LDAP directory set to read-only and try and add a user to a group via the CSUM interface, CSUM tells you that the adding was successful, but the user isn't actually added to the group. On the CSUM side, it probably shouldn't give a success message to the users. On the Confluence side, we should probably let CSUM users know that from 3.5+, they'll have to set their LDAP directories to 'Read-only with Local Groups' if they want to add LDAP members to groups.

The reason it was a success message is that the user already existed. I understand that it should indicate that the user already existed. Please add a bug for this functionality, but I don't think that this would keep people from using the plugin. I might even classify this as a trivial level, since for most users, it won't matter. Either way, please create a CSUM Jira ticket for this if you don't mind.

Finally, would you please tell us when 3.5 will be available for us to develop against? Could we get an early release?

Thanks,
Gary
