Description

Hello,

I found the problem when adding a new group using CSUM plugin.

If there is a login name of the currently logged user in With Users (optional) form input field then the System error will occur.
The only solution is manually remove that user from that group in JIRA.
Then the error will disappear.

Please follow the attached snapshots one by one.
1/ There is no groups in the beginning.
2/ Trying to cretae a group CEITPROJECT-xyz with the user rubringt (currently logged - it's me).
3/ System error occurs.
4/ I have to manually remove the user rubringt from the CEITPROJECT-xyz* group in JIRA
5/ Error disappears. But, the group is not visible in the grouplist of CSUM (I created another issue for this problem - SUSR-128)

I will appreciate any help,
thank you in advance!

Best regards,
Tomas Rubringer

Gary S. Weaver added a comment - 08/Mar/11 7:40 AM - edited

Tomas,

It is possible that this may be related to SUSR-128. Please see that description and workaround first and try not adding users during group addition and then seeing whether a click on link refresh (not a browser refresh) to clear cache after 30 minutes shows the new group.

If the issue is that Confluence is just not recognizing the changes on-time, then perhaps we need an option in CSUM to disable the ability to add members to groups during group addition, or a delay as noted in the other ticket.

Anyway, all that said, this error is with Confluence itself, it appears. Those classes mentioned in the error are not CSUM classes.

If you could do atlassian-user and CSUM debug logging as mentioned in the troubleshooting part of the CSUM wiki page:
https://studio.plugins.atlassian.com/wiki/display/SUSR/Confluence+Space+User+Management+Plugin

then scrub it of private data, and attach it to this ticket, I will add an Atlassian Jira ticket so they are aware of this issue in their API, assuming it is, as I suspect.

Unfortunately, IMO it seems that Jira/Confluence integration the way it was suggested earlier to be done by Atlassian (without Crowd- just by sharing database tables) is flawed. I would think that if you manage users/groups via the Jira SOAP API, the way it should work is that it does not return from that call until it is not only done managing the users/groups in Jira, but also done synchronously telling Confluence that it should refresh its user/groups cache, assuming it is using one. However, I think it is only updating Jira's users/groups and not telling Confluence that it should refresh its users/groups cache, assuming it is using one.

Thanks,
Gary
