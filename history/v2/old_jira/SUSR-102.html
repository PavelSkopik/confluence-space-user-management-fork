Description

We have installed and configured csum plug-in version 2.0.5 in confluence version 2.7.3. User management in confluence is delegated to JIRA which is configured to osusers module for user management. We are happy to have working configuration with us where we are able to create space specific groups and users(new and exisiting), however, we are little concerned by the error message being thrown while adding/creating new user. Let me explain it further.

When a new user is created. it is added successfully to jira-users, confluence-users and wiki-space-specific group, however, at the end of successful create operation, following error is reported back on the page:

User jira-users was not removed from group <account_name> (possibly it was not a member)

where <account_name> is name of new account being created.

I perused the log file and an exception is thrown at JiraSoapUserManagementService.addUsersByUsernameToGroups(JiraSoapUserManagementService.java:146)
and have attached the stack trace for your examination.

As I mentioned that though user creation process is a success, but report of this error confuses wiki administrator and end users enough to come back to us and our main aim to use this plugin is to delegate the user/groups management to wiki space owners. Therefore, any help in fixing this is appreciated.

Thanks much,

Hardeep

Gary S. Weaver added a comment - 09/Nov/09 2:03 PM

Hardeep,

I just looked through the log that you provided and can say a few things that might help or might not:

  • You have Confluence configured to use the OSUSER schema which I know that Jira probably uses as well? If that is the case, you might not need the Jira user management option turned on. Have you tried specifying that you are using Confluence for user management? If not, please try that.
  • The error that you are talking about (rollback) is not a CSUM error, nor is it due to the CSUM plugin from what I can tell. The OSUser repository is not a good choice for Confluence unfortunately, which is why Atlassian stopped using it as the default. OSUser repo does not support lots of users and/or groups for one thing (it can be very slow). If you must continue to use OSUser, you might lookup that error to see if you can track down why it is happening. I've seen it before, but we transitioned to Atlassian schema as soon as we learned about the problems with OSUser schema and since then I don't remember having seen this error on our side (although my memory is not perfect).

The last thing is that another customer of CSUM using the Jira user mgmt functionality had reported previously to us is that they had lag time between operations in the Jira repository and when it showed up in the Confluence repository (or vice versa- I forget). This issue was not with CSUM, but with something else (maybe something was caching?). This made it incredibly difficult for that customer to use the CSUM plugin because the CSUM plugin would add or remove a user/group and then it would be some amount of time before that change was reflected. I remember this being a reason that at least one of their instances of Confluence was transitioned off of Jira for user mgmt.

If you have any control over what your company uses for user management, I'd strongly suggest moving off of something where Confluence and Jira are sharing the user repo via the method that Atlassian described a few years ago. You might talk with Atlassian outside of the context of using this plugin (as they don't support it, although they've recommended it in the past to their customers) and ask what alternatives you might have for user management that might be more well supported and less problematic.

Hope this helps, and we'd be happy to continue to assist on this issue if there is anything specifically in the plugin's code that is not working that we can fix. Right now, unfortunately I am not sure what we should do.

Take care,

Gary

public boolean isMemberOf(String userName, String groupName) {
        log.debug("isMemberOf() called. userName=" + userName + " groupName=" + groupName);
        boolean result = false;
        Group group = userAccessor.getGroup(groupName);
        if (group != null) {
            Pager pager = userAccessor.getMemberNames(group);
            List memberNames = PagerUtils.toList(pager);
            if (memberNames != null) {

if (remoteGroup != null) {
                    if (!isMemberOf(remoteUser.getName(), groupName)) {
                        if ("jira-users".equals(groupName)) {
                            log.debug("We won't add " + remoteUser.getName() + " to group jira-users (SUSR-102).");
                        }
                        else {
                            log.debug("Adding " + remoteUser.getName() + " to Jira group " + groupName);
                            jiraSoapService.addUserToGroup(token, remoteGroup, remoteUser);
                        }
                    }
                    else {
                        log.debug("Did not add user " + remoteUser.getName() + " to group " + groupName + ". It was already a member.");
                    }
                }

and remove it does

if (isMemberOf(remoteUser.getName(), groupName)) {
                        if ("jira-users".equals(groupName)) {
                            log.debug("We will not remove " + remoteUser.getName() + " from group jira-users (SUSR-102).");
                        }
                        else {
                            log.debug("Removing " + remoteUser.getName() + " from Jira group " + groupName);
                            jiraSoapService.removeUserFromGroup(token, remoteGroup, remoteUser);
                        }
                    }
                    else {
                        log.debug("Did not remove user " + remoteUser.getName() + " from group " + groupName + ", because it was not a member of that group.");
                    }
