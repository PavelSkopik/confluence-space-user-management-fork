Description

We use the CSUM plugin version 2.0.5 for managing users and groups in spaces.
Note, that we use LDAP users and non LDAP users.

Filling a group with pure LDAP users shows no problems. E.g. we have a group with currently 131 LDAP users. All users can be displayed by the plugin.
But after adding one more user which comes not via LDAP, the list of all users is suddenly limited to 100. If we remove this user again (via the administration console), the plugin lists all 131 again.

Please find the attachments for details.

Martin,

We use the Confluence User paging API supplied by Confluence and get the list of users and each page of users from that API (and the numbering of users also comes from the API/is derived from the API results, I believe).

When you say the users are in LDAP, you mean that the users in the ecoit-awts group were added via the CSUM plugin (via the text-field it allows you to enter a list of user IDs into), and that you used the CSUM plugin's LDAP integration functionality that in-turn uses the LDAP user repository configured as the second user repository in atlassian-user.xml?

If you click on the refresh link in the page (part of the CSUM plugin page), does the number of users correct itself? If you completely quit the browser and open up a new one in a new user session in Confluence, is the number of users still incorrect after the addition of the new user?

If you use Confluence's own group administration area, and you add a user to that group there, do you see a similar bug?

This is a new issue to me, but we do not use Confluence 2.10.3. You might want to look at the Confluence release notes for versions of Confluence after 2.10.3 to see whether any bugs were fixed related to users, groups, paging, repositories, or LDAP integration.

We will do our best to help you to get this issue resolved, but if it turns out to be a bug with Confluence (hopefully that is shown in its group administration area also), then you'll want to contact Atlassian support (and do not mention this plugin at all, as they do not support it). If it turns out to be a bug in the Confluence API that only shows up in the CSUM plugin, and it isn't fixed in a later version of Confluence, then we might have to submit a ticket and hope that they work on it. (If it isn't affecting Confluence or a Confluence-supported plugin, I'm not sure whether it would be prioritized. The user/group pager in the API is really not well documented, nor used much with the exception of this plugin and Confluence's own user/group administration area, at least as far as I know. We are using it, because we have to in order to easily support various Confluence repositories (OSUser, Atlassian-user) and limit the time required to do lookup for the result set pages (at least in Atlassian-user; OSUser didn't support true paging, so it is slow with lots of users, but that is only used in much older versions of Confluence, pre-2.6).

Take care,
Gary

Gary S. Weaver added a comment - 06/Aug/10 5:49 AM

Richard Atkins Atlassian updated CSP-49519:
---------------------------------------------

Status: Waiting For Customer (was: Developer Review)

Hi Gary,

I have the reasons the values from these utility classes are changing like this:

  1. Before the new user is added, all users must be being loaded through a single Pager such as DefaultPager, which has no upper limit on the number of entries it will load in a single page (that is, a List of entries), or one of the AbstractLDAPPager implementations, which will load only 100 entries per page, but will iterate quite merrily past that limit (causing additional LDAP data dips every 100 entries).
  2. After the new user is added, the users are then loaded through MergedListPager, which (like all AbstractLDAPPager implementations) is limited to 100 entries in a single page, however unlike all the other Pagers, this implementation does not automatically load more entries during iteration. This would explain why you're seeing some sizes drop from 131 to 100.
  3. If MergedListPager exhausts one page in the first Pager its using, it doesn't automatically move on and try to view the contents of another page from the same Pager, or from any another Pager it knows about. If the hibernate user/group list was used first, this would explain why you're seeing the size drop from 131 to 0 or 1.

The reason the MergedListPager doesn't go and load more entries from the current Pager is so that it doesn't block waiting on a possibly expensive call to the underlying Pager. However, it's a known defect that it doesn't automatically move on to the next Pager if the current Pager returns true to onLastPage(), and the next Pager already has entries loaded to show. See http://jira.atlassian.com/browse/CONF-5106 for details, and to vote and watch for changes to this issue.

In the meantime, your plugin needs to call Pager.onLastPage() to check if the Pager that PagerPaginationSupport and PagerUtils are reading behind the scenes has more pages to show you, and if so, allow the user to browse beyond the currently known maximimum number of entries (e.g. by showing "11..20 of more than 100"). It also needs to invoke Pager.nextPage() to trigger loading more entries. Once the MergedListPager loads the next page, the size returned by either of the helper classes will increase by however many entries that next page contains.

Richard
