Gary S. Weaver added a comment - 26/Jun/09 3:14 PM

Reopened.

Aditee, the refresh link is in the "Manage Users/Groups" tab in browse space at the bottom:

Bulk Change Tool | Edit Space Permissions | Refresh <--- this is the one

Note that this is not the first time that Cisco has reported similar issues (although not exactly the same).

Raju Kadam of Cisco had issues like that previously and iirc determined that there was a delay on the Cisco side for changes to users and groups because of the way things were setup. While the setup you are working on may be newer and not using Jira integration for users/groups anymore, no other users have reported issues with a delay and we haven't either. That's not to say there is nothing that can be done in the plugin or in Confluence API but if there is, I don't know what it is.

It is true that the plugin caches users and groups, and this could definitely be related to the issue if the Confluence API is not actually returning that the users/groups have changed after the plugin calls the Confluence API to add/remove/change them.

However, not caching would not work in your environment because you have too many users/groups to do that, from what Raju said.

So if the Confluence API that the plugin is using is not reporting that the users/groups have changed when changes are made, then I'd say the issue may be neither with the plugin nor with Confluence but rather with the user/groups repository or the connector to it. That might be a Crowd issue in your case (I think Raju said you all were using Crowd) or related to some kind of configuration change that could be made on your side with Crowd. If not, then Atlassian could perhaps try putting in some sort of delay before it checked the user repository (and make that delay configurable). Possibly that ability is already there. You could ask for assistance with Atlassian on trying to configure Confluence/Crowd such that it has a better chance of getting all of the user/group info back.

None of this should happen obviously. The Confluence API should return the correct users and groups after any changes to them. So right off hand, I'd say that you should persue trying to replicate this issue by writing a plugin that just tries to delete/add users and then makes sure that it can get the full list back each time, and perhaps if it only happens sporatically, maybe you would need to cron a test that did it and save those specific logs. But if you can't do that, then maybe just asking Atlassian for assistance in configuration to make sure that the API can get all users/groups back would be better?

I'd like to help, but the best I think that we'd be able to do would be if you sent us a debug log of just the issue where you have debug logging for the UserAccessor, SpacePermissionManager, the pager classes, and anything else related to getting or paging through the data, then maybe we could assist you in creating a Jira ticket with Atlassian, but that is about all we could do without knowing more.

Thanks,

Gary
