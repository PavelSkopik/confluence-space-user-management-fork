Description

They click on Submit and receive an error message saying, "User was not removed from the group (possibly it was not a member)". Only Confluence admins can add users. All our Space Admins are reporting the same problem.

Debbie,

Thanks for the info.

From what Ben Speakmon of Atlassian said (although I haven't confirmed it yet), Confluence 2.10 and later do not support managing group membership by a space administrator. By this I don't mean just that the plugin doesn't let you- I mean that the Atlassian Confluence API doesn't let you- at least, not in the way that it did before (and I don't know the "new" way yet that Ben alluded to).

If the version of Confluence is less than 2.10, the problem we are talking about should not occur. In versions of Confluence 2.10 or greater, if the user is a confluence administrator, they will not see the issue, but if they are only a space administrator, they would see the issue.

I've only confirmed this in Confluence 3.4.6 (current version). I've not tried to reproduce in other versions.

If you are sure that what I'm saying is incorrect please let me know, as it will help expedite a possible remedy.

If Ben doesn't come up with a solution, it could mean that we would need to modify the CSUM plugin to use an alternative Confluence API (remote or SOAP), assuming they allow us to do what we need to. And if that doesn't work- who knows. Maybe we'd have to modify the DB directly, but we'd run into all sorts of complexity if we went down that path.

The current workaround is (and I don't think this is feasible for most people) to give confluence administrator rights to the space admins or revert to Confluence 2.9.x (which is almost certainly not feasible for most people unless they can just delay/abandon their Confluence upgrade from pre-2.10).

We'll keep working on it, and please keep providing as much info as you can that you think will help.

Gary

Matt Ryall [Atlassian] added a comment - 16/Jan/11 4:50 PM
Hi Gary,

The UserAccessor in Confluence fits into our service layer, meaning it is responsible for permission checks as well as performing the actual changes. I'm not sure about Confluence 3.1 or 3.4, but some permission checking has been in place in UserAccessor.addMembership() since Confluence 2.7. In that release, we added a new layer of administrative permission and it was necessary to ensure the UserAccessor checked the group membership of the person doing the change before acting on it.

To do user management changes without permission checks in all Confluence versions up to 3.4.x, you can use the Atlassian-User managers directly: com.atlassian.user.GroupManager and com.atlassian.user.UserManager. Note that these methods will not fire events or perform other Confleunce-specific functionality which the UserAccessor normally takes care of.

From Confluence 3.5, there will be a new low-level API for user management that will be documented when we release it. The interface is called CrowdService.

Regards,
Matt
