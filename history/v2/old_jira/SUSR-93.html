Description

Alwyn Wong of Customware researched a slowness issue with Crowd found by Tim Colson of Cisco with the CSUM plugin and determined the following:

----Original Message----
From: Alwyn Wong
Sent: Friday, April 03, 2009 1:35 AM
To: Tim Colson
Cc: James Matheson
Subject: RE: Support help with CSUM

Hi Tim,

Here are the results of my investigation.

I do apologize in advance that this is not a comprehensive report due our
limited time.

But here are my findings:

I have pin pointed the location and method that is causing this slow
performance problem and it begins with the doUserSearch() method.
That method then goes away and attempts to find users based on a query.
The query it constructs is then passed to userAccessor class from
Confluence and will retrieve a searchResult which are all valid API calls.
From that point onwards it delegates the responsibility to Confluence to
handle the query and there is very little we can do from that point.

So the question is do we need this "expensive" query? From our limited
time of investigation I can make that judgment as it requires much more
investigation to give a definite answer.

There is also a comment in the code which I copied below:
//TODO: this is really slow with osuser search. must use
http://confluence.atlassian.com/display/DOC/How+to+Improve+User+Search+Performance
But this may not be relevant to you as you are using Confluence 2.10 but
I'll point it out anyway.
I mention that this only occurs after the crowd integration so I compared
the two search results and queries. The queries were the same (as
expected) but search results where opposite. i.e Crowd integration
returned all users and without crowd it returns zero users.

Here is the method that I believe is slowing the system:

UserNameTermQuery query = new
UserNameTermQuery(advancedUserQuery.getPartialSearchTerm(),
advancedUserQuery.getSubstringMatchType());
SearchResult result = userAccessor.findUsers(query);

What it tries to do is search for users based on a query and the query
object is of type UserNameTermQuery. The UserNameTermQuery object
accepts two parameters which are:
1. search term
2. match type.
The match type is "startwith" which i don't see anything wrong with but
the search term is undefined (i.e null ) and before crowd integration the
result of this query will be zero users, oppose to after the crowd
integration it will grab ALL the users and hence the slow.

Obviously we can omit this call to improve the performance but I do not
recommend this because there will be consequences. I couldn't dig far
enough to let you know the consequences but it's logical it assume there
are potential problems especially if we try to just blindly skip this
method call without a more detailed investigation.

If you want this performance enhancement I recommend to organize a support
contract with James Matheson(CustomWare's Atlassian Principle) for the
CSUM plugin so we can investigate further and recommend a viable solution
oppose to a temporarily solution and hope that crowd will fix this
problem.

I look forward to your thoughts.

Regards,
Alwyn Wong
